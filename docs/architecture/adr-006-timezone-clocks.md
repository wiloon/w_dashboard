# ADR-006: 多时区时钟——基于 IANA 时区本地计算

## Status

Accepted

## Context

**问题**：需要展示北京时间、纽约时间，以及任意可配置时区的时间。

**约束**：
- 纯本地应用，无后端。
- 时间显示需随系统时钟实时更新，并正确处理夏令时（DST）。

## Options Considered

| 方案 | Pros | Cons | 何时合适 |
|------|------|------|----------|
| A. 本地基于 IANA 时区数据库计算 | 离线可用；正确处理 DST；零依赖外部服务 | 依赖系统/库的 tz 数据 | 总是 |
| B. 调用在线世界时钟 API | 不依赖本地 tz 数据 | 需联网；纯计算却引入网络依赖，不合理 | 几乎从不 |

## Decision

**选择方案 A**：在客户端基于 IANA 时区（如 `Asia/Shanghai`、`America/New_York`）本地计算各时区当前时间，按秒/分刷新。

实现归属：时钟是纯展示逻辑且各端 UI 框架都有成熟的日期时间能力，因此**时钟由各端 UI 自行实现**；时钟列表（label + IANA tz id）由配置规格统一定义，两端按同一规格读取。

## Rationale

1. 纯计算无需联网，方案 B 为纯计算引入网络依赖不合理。
2. IANA 时区数据库正确处理夏令时与历史偏移。
3. 时钟刷新与渲染强依赖各端 UI 的事件循环/定时器，放在 UI 层最自然简单。

## Trade-offs

- 时钟逻辑在两端各写一份（极薄，仅"按 tz 格式化当前时间 + 定时刷新"）。

## Consequences

- **Positive**：离线可用、DST 正确、实现简单。
- **Negative**：依赖运行环境的时区数据；时钟代码两端各一份。
- **Mitigation**：配置规格统一定义时钟（label/tz）并要求校验 tz id 合法性，确保两端配置来源一致；UI 仅做格式化与刷新。

## Revisit Trigger

- 时钟逻辑变复杂（如世界时区地图、日出日落等），值得在规格中统一定义并由两端一致实现。
