# ADR-002: 代码复用——规格驱动，两端独立原生实现

## Status

Accepted（修订自原决策"共享 Rust core + UniFFI"，详见末尾「修订说明」）

## Context

**问题**：Linux 与 macOS 两端都要原生体验（Linux 用 Rust + Iced/libcosmic，macOS 用 SwiftUI）。
git 同步判断、chezmoi 状态解析、天气解析、配置加载等核心逻辑如何在两端保持一致，而又不牺牲各端的原生质感？

**约束 / 前提**：
- macOS 要求地道的原生体验（菜单栏、系统集成、观感），用 SwiftUI。
- Linux 用 Rust 生态的 Iced / libcosmic。
- **本项目处于「规格驱动 + AI 生成」的工作模式**：用 ADR + SDD + task-spec 精确定义软件，平台代码由 AI 生成；需求变更时先改规格，再由 AI 重新生成/同步各端代码（见 ADR-008）。
- 业务逻辑本身复杂度有限：核心是「调用 `git`/`chezmoi` 命令 + 解析机器可读输出 + 派生状态 + 调天气 API」，不属于算法密集型。

## Options Considered

| 方案 | Pros | Cons | 复杂度 | 何时合适 |
|------|------|------|--------|----------|
| A. 两端完全独立实现，靠规格 + 测试向量保证一致 | 两端 100% 原生地道；无 FFI、无跨语言构建链；AI 生成单一语言代码最顺 | 逻辑实现两遍；一致性需靠规格精度 + 共享测试向量来保证 | 中 | 规格驱动、逻辑复杂度可控、要求各端极致原生 |
| B. 共享 Rust core + UniFFI 暴露给 Swift | 逻辑只编译一份，编译期强一致 | 需搭 Rust↔Swift FFI 桥；交叉编译 + XCFramework 打包；core API 受 UniFFI 类型限制（时间只能 i64、错误只能 enum 等）；mac 侧不够地道 | 高 | 逻辑复杂、改动频繁，且能接受 FFI 成本 |
| C. 两端都用 Rust/Iced | 框架与逻辑全共享 | macOS 上 Iced 不如 SwiftUI 原生 | 低 | 不在意 mac 原生体验 |

## Decision

**选择方案 A**：不设共享的二进制 core，**Linux 与 macOS 各自用最地道的原生技术独立实现全部逻辑与 UI**。

一致性不再依赖「同一份编译产物」，而是依赖：

1. **精确、语言无关的行为规范**——数据模型、状态派生规则、解析规则写到无歧义（见 SDD §5/§7）。
2. **共享的「行为契约 / 测试向量」**——用纯数据（JSON）定义一组 `输入 → 期望输出` 用例，两端各自的测试都喂同一份向量、必须得到同一结果（见 SDD §10）。

这样「单一事实来源」从代码层上移到规格层：规格是源，两端代码都是规格的投影。

## Rationale

1. **契合规格驱动模式**：既然代码由 AI 从规格生成，再生成单一语言的地道代码，比生成「受 FFI 约束的 Rust + 绑定胶水」更可靠、产物更干净。
2. **两端 100% 原生**：Linux 用 Rust 最地道的写法，macOS 用 Swift 最地道的写法，没有任何一端为迁就跨语言而妥协（排除方案 C，规避方案 B 的 mac 侧妥协）。
3. **彻底消除 FFI 成本**：不需要搭 FFI 桥、不需要交叉编译 XCFramework、不受 UniFFI 类型限制——各端可用最自然的类型（Swift 用 `Date`，Rust 用 `chrono` 等）。
4. **重复成本可接受**：核心逻辑是「调命令 + 解析文本 + 派生状态」，用两种语言各写一遍的成本不高，且能被 AI 生成与共享测试向量很好地兜住。

## Trade-offs

- **逻辑实现两遍**：放弃方案 B 的「编译期强一致」，改由规格精度 + 测试向量保证一致。
- **一致性责任转移**：从编译器转移到「规格 + AI + 测试」。这要求规格必须足够精确，且测试向量必须覆盖关键边界，否则两端会在细节上悄悄分叉（如 porcelain 罕见标志、边界态的 `RepoState` 判定、天气缺字段降级）。

## Consequences

- **Positive**：两端极致原生；零 FFI / 零跨语言构建复杂度；AI 生成顺畅；各端可独立演进、可并行开发。
- **Negative**：存在两份实现，若规格不精确或测试向量不足，可能行为漂移。
- **Mitigation**：
  - SDD 把状态派生规则写到「给定 X 必出 Y」的无歧义程度。
  - 维护一份语言无关的测试向量（golden test vectors），**两端 CI 必须用同一份向量验证**，作为一致性的权威。
  - 把「采集（有副作用、不确定）」与「解析 + 派生（纯函数、确定）」分层，测试向量锁定纯函数层；真实联网/调命令只做少量 e2e 冒烟。
  - 任何影响行为的规格变更，必须同时更新测试向量，再让两端同步。

## Revisit Trigger

- 业务逻辑变得足够复杂或改动极频繁，使「实现两遍 + 维护测试向量」的成本超过 FFI 方案。
- 两端频繁出现一致性漂移且测试向量难以覆盖。

## 修订说明

本 ADR 原先选择「共享 Rust core，macOS 经 UniFFI 调用」（方案 B），以「编译期只有一份逻辑」来保证一致性。
后因项目确立**规格驱动 + AI 生成**的工作模式（ADR-008），改为方案 A：以规格为单一事实来源、两端独立原生实现、用行为契约 + 测试向量保证一致。
取舍的本质是：**把工程复杂度（FFI / 跨语言构建链）换成规格纪律（精确规范 + 共享测试向量）**。
