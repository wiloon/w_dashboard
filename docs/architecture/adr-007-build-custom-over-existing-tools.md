# ADR-007: 自建统一原生面板，而非复用 gita/mani 等现有工具

## Status

Accepted

## Context

**问题**：git 多仓库状态、chezmoi 同步状态这两类需求，已有成熟工具能覆盖大部分：

- 多仓库 git 状态：`gita`（`gita ll` 直接彩色标出 dirty/ahead/behind）、`mani`（配置驱动、`mani run -a git status`）、`myrepos (mr)`，或一个简单 shell 循环。
- chezmoi 同步：chezmoi 自带 `chezmoi status` / `chezmoi git -- status` / `chezmoi doctor` / `chezmoi verify`。

也就是说，"出门前检查家里 PC 是否所有仓库都已提交/推送 + chezmoi 是否同步"这一刚需，约等于 `gita ll` + `chezmoi status` 两条命令，或一个几十行脚本即可满足。唯独"天气 + 多时区时钟 + 上述状态揉进一个统一图形面板"没有现成品。

**约束 / 取向**：作者**明确希望拥有一个统一、可控、自己掌握的原生面板**，并接受其中一部分功能与现有工具重叠。该项目的价值不止于"解决刚需"，也包含掌控感、长期可演进性，以及借此实践 Rust/Iced/SwiftUI 等技术栈。

## Options Considered

| 方案 | Pros | Cons | 何时合适 |
|------|------|------|----------|
| A. 最小脚本（封装 gita/mani + chezmoi status + 通知） | 几小时即可、零维护、复用成熟工具 | 无统一图形面板；天气/时钟需另寻；可定制性低；不掌控实现；**需记忆多个工具的命令与用法** | 只想最快满足刚需 |
| B. 做原生面板，但调用并解析 gita/chezmoi 输出 | 少造轮子；聚焦 UI 整合 | 受外部工具输出格式/演进约束；多一层依赖；掌控度居中 | 想要面板又想省采集逻辑 |
| C. 自建统一原生面板，自实现采集逻辑 | 完全掌控；无外部 CLI 依赖；可深度定制与演进；技术实践价值；**打开即看、零命令记忆负担** | 明知与现有工具部分重叠（造轮子）；开发量最大 | 看重掌控感、统一体验与长期演进 |

## Decision

**选择方案 C**：自建统一原生面板，git/chezmoi 状态由应用自行采集（按 ADR-003 通过子进程调用 `git`/`chezmoi` 原生命令，但不依赖 `gita`/`mani` 这类第三方聚合工具）。**明确接受这部分功能与现有工具存在重叠。**

## Rationale

1. 作者的目标不只是"解决出门前检查"这一刚需，而是要一个**统一、可控、自己掌握**的原生面板——这是现有零散工具无法提供的整体体验。
2. 不依赖 `gita`/`mani` 的输出格式，避免被其演进或解析差异绑架，掌控度最高（区别于方案 B）。
3. **降低记忆负担**：复用多个零散工具，意味着要记住 `gita`/`mani`/`chezmoi` 各自的命令、参数与输出方式；一个统一的原生面板把这些收敛成"打开即看"，不需要记忆任何命令。这是统一面板相对"工具拼凑"的实质性体验优势。
4. 该项目本身带有技术实践与长期演进价值，"造轮子"在此是经过权衡的、有意识的选择，而非疏忽。
5. 仍遵循 ADR-003：直接调用 `git`/`chezmoi` 这两个底层工具是合理的（它们是事实标准、且 chezmoi 逻辑本就无可替代），不属于"重复造轮子"的范畴；不复用的是上层聚合器。

## Trade-offs

- 接受比"最小脚本"大得多的开发量，以及与现有工具的功能重叠。
- 放弃方案 A 的"几小时见效、零维护"。

## Consequences

- **Positive**：完全掌控数据采集与展示；统一整合天气/时钟/git/chezmoi；无第三方聚合 CLI 依赖；可按需深度演进。
- **Negative**：投入大；重复实现了部分已有能力。
- **Mitigation**：用 YAGNI 约束范围（见 SDD §11 非目标），不让"造轮子"扩散成"造大而全的轮子"；两端直接复用 `git`/`chezmoi` 底层命令而非从零实现 git 协议，降低不必要的重复。

## Revisit Trigger

- 自实现采集的维护成本明显超出收益，且作者不再看重"完全掌控"这一点。
- 出现某个现成工具已能提供完全等价的统一面板体验。
