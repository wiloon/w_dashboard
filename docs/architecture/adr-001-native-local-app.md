

















# ADR-001: 应用形态——纯原生本地应用，不做后端

## Status

Accepted

## Context

**问题**：需要一个 dashboard，展示 git 仓库同步状态、chezmoi 配置同步状态、多时区时间、天气。
其中 git / chezmoi 状态的数据**天然只存在于每台主机的本地磁盘上**，任何不在该机器上运行的进程都无法直接读取。

**约束**：

- 使用者有两台机器：Beelink SER8（家里的 Linux 桌面）、MacBook Air M2（外出时使用）。
- 关键澄清：**每台机器只需看本机的状态**，不需要在一侧看到另一侧的状态。
- MacBook 会漫游，常处于 NAT 后、IP 不固定，外部无法主动连接它。
- 优先低维护成本、低运行依赖。

## Options Considered


| 方案                            | Pros                           | Cons                 | 复杂度 | 何时合适         |
| ----------------------------- | ------------------------------ | -------------------- | --- | ------------ |
| A. 纯原生本地应用，无后端                | 自包含；无需部署/运维；无网络依赖（除天气）；本地读盘最直接 | 无法跨机汇总；天气/时区逻辑各端自理   | 低   | 只看本机状态时      |
| B. 轻量 Agent + k8s 中心收集器（Push） | 可统一看多机；手机可看                    | 需后端、存储、鉴权、部署；运维成本高   | 高   | 需要跨机汇总/远程查看时 |
| C. 浏览器 / WASM 直接读本地           | 无需安装                           | 技术上不可行（沙箱无文件系统与进程执行） | —   | 永不           |


## Decision

**选择方案 A**：在每台主机上运行一个原生本地应用，只展示**本机**的状态。不引入任何后端、数据库或网络服务（天气除外，直接调公网 API）。

## Rationale

1. 需求已明确为"只看本机"，方案 B 的跨机汇总能力属于不需要的功能，为它引入后端是过度设计（YAGNI）。
2. MacBook 漫游 + NAT 使得"中心主动拉取"不可行；而本地应用读本机磁盘没有任何网络可达性问题。
3. 方案 A 自包含，无部署与运维负担，符合"低维护成本"约束。
4. 方案 C 在浏览器/WASM 沙箱下无法读取任意本地路径、无法执行 `git`/`chezmoi`，直接排除。

## Trade-offs

- 放弃"在一个面板里同时看到两台机器"的能力。
- 放弃"用手机/任意浏览器远程查看"的能力。
- 天气、时区等与机器无关的逻辑无法通过后端集中提供，需要在客户端处理（影响很小）。

## Consequences

- **Positive**：系统极简、零运维、启动即用、无网络可达性问题。
- **Negative**：未来若想跨机汇总或远程查看，方案 A 本身不支持。
- **Mitigation**：两端都将"状态采集逻辑"与"UI"分层解耦（见 ADR-002），采集层产出结构化数据。未来若需上报，只需在采集层之外各加一层上报模块，UI 与采集逻辑不动，即可平滑演进到方案 B。

## Revisit Trigger

- 出现第三台及以上机器，或确实需要在手机/远程统一查看多机状态。
- 需要历史趋势、告警等只能由常驻服务提供的能力。
